1. Introduction
1.1 Overall Description
Secure image transfer plays a vital role in ensuring the confidentiality and integrity of digital visual
data, especially in today's interconnected world where images are frequently shared across networks.
With the increasing risk of interception, tampering, and unauthorized access during image transmission
over email and other communication channels, protecting images from malicious actors is of critical
importance. Advanced Encryption Standard (AES) with a 256-bit key provides one of the strongest
symmetric encryption mechanisms available today, making it the ideal choice for securing image data.
The main purpose of this project is to develop a Secure Image Transfer System — a Python-based
desktop application that encrypts image files using AES-256 encryption in CBC (Cipher Block
Chaining) mode before transmitting them securely via email. The system features a modern dark-mode
graphical user interface built with Tkinter, complete with email sending (SMTP over SSL) and
receiving (IMAP) capabilities. The receiver can download the encrypted .enc file from their email and
decrypt it locally using the same secret key. This ensures that even if the email is intercepted during
transit, the image data remains completely unreadable and protected. The project demonstrates how
cryptography, combined with secure communication protocols, enhances end-to-end data privacy for
image transfer.
2. Existing System
In the existing system, images are frequently shared via email and messaging platforms without any
encryption or security measures. The image files travel in plaintext form, making them vulnerable to
interception by man-in-the-middle attacks during network transmission. Some systems use basic or
weak encryption techniques such as simple XOR-based pixel manipulation or Caesar cipher-like
substitution methods, which can be easily broken using modern computational tools and brute-force
attacks. Most systems do not implement proper key generation or derivation, relying on short or
predictable passwords. Additionally, there is no integrated mechanism where encryption and email
transmission happen in a single workflow — users must manually encrypt the file using one tool and
then attach it to an email separately, leading to user errors and security gaps. Most existing tools lack a
graphical interface, requiring command-line usage and providing no visual feedback on the encryption
or decryption process. There is also no mechanism to ensure the image has not been corrupted or
tampered with during the encryption-decryption cycle. Hence, the current systems fail to guarantee
comprehensive data security, user convenience, and reliable end-to-end image protection.
3. Proposed System
The proposed Secure Image Transfer System overcomes all the drawbacks of the existing system by
providing an integrated, end-to-end secure image transfer solution. It implements AES-256 encryption
in CBC mode with a randomly generated 16-byte Initialization Vector (IV) for each encryption
operation, ensuring that encrypting the same image twice produces entirely different ciphertexts.
User-provided passwords are securely hashed using SHA-256, producing a deterministic 256-bit key
suitable for AES-256 encryption. Users can also generate a cryptographically strong random key using
the built-in key generator. The system implements PKCS7 padding to ensure that the image data aligns
perfectly with the 16-byte AES block size requirement. The system encrypts raw image file bytes
directly, not pixel values, ensuring bit-perfect recovery of the original image after decryption with zero
quality loss. After encryption, the encrypted .enc file can be sent directly to a recipient via a secure
SMTP SSL connection with the encryption key and decryption instructions included in a professionally
formatted HTML email. The receiver can use the built-in email checker to connect to their inbox via
IMAP, automatically search for and download .enc attachments. A clean, intuitive Tkinter-based
graphical interface with a dark theme, card-based layout, progress indicators, and status messages
ensures an excellent user experience. The system also features secure key generation, clipboard
support, and comprehensive error handling. Thus, the proposed system provides a high level of data
protection, reliability, convenience, and a complete end-to-end secure workflow for image transfer.

5. Implementation
The implementation is carried out using Python 3 with the pycryptodome, Pillow, and Tkinter
libraries. The AES-256 algorithm operates on the raw image file bytes, ensuring confidentiality and
lossless recovery. The system follows a modular architecture with clear separation of concerns, where
each module handles a specific responsibility. Modules are implemented for encryption, decryption,
key generation, email sending, email receiving, GUI, environment configuration, and utility functions.
5.1 Module Description
5.1.1 AES-256 Encryption Module (encryptor.py)
This is the core module responsible for encrypting and decrypting image files using AES-256 in CBC
mode. The encrypt_image function reads raw image bytes, builds a plaintext payload with an embedded
file extension header using magic bytes SITS1 plus extension metadata plus image data, generates a
random 16-byte IV, applies PKCS7 padding, encrypts using AES-256-CBC, and writes the IV plus
ciphertext to a .enc file. The decrypt_image function reads the encrypted file, extracts the IV which is
the first 16 bytes and the ciphertext, decrypts using AES-256-CBC, removes PKCS7 padding, parses
the embedded extension metadata, and writes the restored image bytes to the output file with the correct
extension. It also implements both _pkcs7_pad and _pkcs7_unpad functions to ensure data alignment
with the 16-byte AES block size.
5.1.2 AES-256 Decryption Module (decryptor.py)
This module provides the public decryption API. It delegates to the core decrypt_image function in
encryptor.py to maintain centralized encryption and decryption logic. It automatically determines the
output file path by stripping the .enc extension from the input filename.
5.1.3 Key Generation and Management (key_manager.py)
This module handles all key-related operations. The generate_key function generates a
cryptographically strong 256-bit random key using os.urandom(32) and returns it as a URL-safe
Base64-encoded string for easy copy and paste. The hash_key function derives a 256-bit AES key from
a user-provided password string using SHA-256 hashing, producing exactly 32 bytes suitable for
AES-256 encryption.
5.1.4 Email Sending Module (email_sender.py)
This module handles secure email transmission of encrypted image files using SMTP over SSL. It
connects to the SMTP server on port 465 using SSL encryption. It constructs both a plain-text fallback
and a beautifully formatted HTML email body containing the encrypted filename, the encryption
password, file size information, step-by-step decryption instructions, and security warnings. The
encrypted .enc file is attached as an application/octet-stream MIME type.
5.1.5 Email Receiving Module (email_receiver.py)
This module enables downloading encrypted attachments from email using IMAP. It connects to the
IMAP server using SSL with configurable timeout. It searches multiple mailbox candidates including
INBOX, All Mail, and Sent Mail for messages containing .enc attachments. It downloads .enc
attachments to a user-specified directory, automatically handling filename conflicts by appending a
counter. It properly decodes MIME-encoded filenames and handles various email encoding schemes.
5.1.6 GUI Module (gui.py)
This is the main graphical interface module built with Tkinter and ttk. It uses a custom dark color
palette with the clam ttk theme. The interface is organized into four distinct cards: Image File Card for
image selection with file validation, Secret Key Card for key entry, generation, and clipboard copy,
Email Settings Card for sender and receiver email configuration with .env persistence, and Actions
Card for encrypt, send, and decrypt operations. Heavy operations run in background threads to keep
the GUI responsive. A progress bar and status label provide real-time feedback on ongoing operations.
5.1.7 Environment Configuration Module (env_config.py)
This module manages persistent email settings using a .env file. It handles loading by parsing the .env
file with support for comments, quoted values, and whitespace. It handles saving by merging new
settings with existing values and writing them back in a clean format. It manages SENDER_EMAIL,
SENDER_APP_PASSWORD, and RECEIVER_EMAIL settings.
5.1.8 Utility and Validation Module (utils.py)
This module provides shared utility functions including logging setup that configures application-wide
logging to both a file and the console, image validation using Pillow's Image.verify to validate that
selected files are genuine readable images, directory management to ensure required directories exist,
and resource path resolution for both source and PyInstaller-bundled environments.
5.2 System Architecture
The system architecture follows a layered modular design. The presentation layer consists of the
Tkinter GUI with dark-mode themed cards. The application logic layer contains the encryption module,
decryption module, email sender, email receiver, key manager, and environment configuration module.
The utility layer provides logging, validation, and helper functions. The foundation layer uses external
libraries including pycryptodome, Pillow, smtplib, imaplib, and standard OS modules. All modules
interact securely, with the GUI layer coordinating operations between the encryption, email, and
utility modules. Background threading ensures non-blocking execution of heavy tasks.
6. Algorithm Implementation
6.1 AES-256 Algorithm (CBC Mode)
The Advanced Encryption Standard (AES) is a symmetric block cipher standardized by NIST. In this
project, AES is used with a 256-bit key and operates in CBC (Cipher Block Chaining) mode. The key
preparation step involves hashing the user's password using SHA-256 to produce a 32-byte key. A
cryptographically random 16-byte Initialization Vector (IV) is generated for each encryption operation.
The original image file extension is embedded in the plaintext payload using a custom header format
with magic bytes. PKCS7 padding is applied to ensure the plaintext length is a multiple of 16 bytes.
The AES cipher then encrypts the padded plaintext in CBC mode, where each plaintext block is XORed
with the previous ciphertext block before encryption. The IV is prepended to the ciphertext and saved
as a .enc file. AES-256 uses a block size of 128 bits, a key size of 256 bits, and performs 14 rounds of
substitution, shift rows, mix columns, and add round key operations. AES-256 is considered
computationally infeasible to break with current technology, as a brute-force attack would require 2
raised to the power of 256 attempts. In this project, Python's pycryptodome library is used for the
implementation. AES ensures that even small changes in input produce drastically different outputs,
guaranteeing strong protection.
6.2 SHA-256 Key Derivation
SHA-256 (Secure Hash Algorithm 256-bit) is used to derive the AES encryption key from the user's
password. The password string is first encoded to UTF-8 bytes, then hashed using SHA-256, producing
exactly 32 bytes which serve as the 256-bit AES key. This ensures that any password of any length
produces exactly 32 bytes, the same password always produces the same key, small changes in the
password produce completely different keys due to the avalanche effect, and it is computationally
infeasible to reverse the hash back to the password.
6.3 PKCS7 Padding
PKCS7 padding is applied to ensure the plaintext data aligns with the 16-byte AES block size. If the
data is 13 bytes, 3 bytes of padding with value 0x03 are added. If the data is exactly 16 bytes, 16 bytes
of padding with value 0x10 are added. During decryption, the padding is validated and removed to
restore the original data exactly.

8. Requirement Specification
8.1 Functional Requirements
The system must support image selection and validation for formats including PNG, JPG, JPEG, BMP,
GIF, TIFF, and WEBP. It must encrypt images using AES-256-CBC with random IV and PKCS7
padding. It must decrypt .enc files back to original images with lossless quality. It must generate
cryptographically strong 256-bit keys in URL-safe Base64 format. It must derive AES-256 keys from
passwords using SHA-256. It must send encrypted files via SMTP SSL with rich HTML email body. It
must download .enc attachments from email inbox via IMAP. It must save and load email settings from
a .env file. It must display encryption and decryption progress with a status bar. It must handle invalid
files, wrong keys, and email errors gracefully. It must support copying generated keys to clipboard and
toggling password field visibility.
8.2 Software Requirements
Python 3.8 or higher is required. The pycryptodome library is needed for AES encryption. The Pillow
library is needed for image validation. NumPy is needed for numerical operations. Tkinter is built-in
with Python and is used for the GUI framework. The smtplib module is built-in with Python for SMTP
email. The imaplib module is built-in with Python for IMAP email. The hashlib module is built-in with
Python for SHA-256 hashing. pip is used as the package manager for installing dependencies.
8.3 Operating Systems Supported
Windows 10 and Windows 11 are fully supported as the primary platform. macOS is fully supported.
Linux distributions including Ubuntu and Fedora are fully supported. The application is cross-platform
as it uses Python with Tkinter, which is available on all major operating systems.
8.4 Technologies and Languages Used
Python 3 is the primary programming language. AES-256 from pycryptodome is used for image
encryption and decryption. SHA-256 from hashlib is used for password-to-key derivation. Tkinter with
ttk is used for the graphical user interface. SMTP over SSL from smtplib is used for secure email
sending. IMAP over SSL from imaplib is used for secure email receiving. Pillow is used for image file
validation. The .env configuration format is used for persistent email settings storage.
8.5 Hardware Requirements
A dual-core CPU such as Intel i3 or equivalent is the minimum processor requirement. 4 GB RAM is
the minimum with 8 GB recommended. 500 MB of disk space is needed for Python, libraries, and
image processing. An internet connection is required for email functionality. A minimum display
resolution of 1024 by 768 is needed. A keyboard and mouse are required for input.
9. System Testing
9.1 Types of Testing
9.1.1 Unit Testing
Individual modules were tested independently. The encryptor.py module was tested for encryption and
decryption of various image formats including PNG, JPG, BMP, GIF, and TIFF with different key
lengths. The PKCS7 padding and unpadding logic was verified with edge cases. The key_manager.py

for .env file parsing with comments, quoted values, special characters, and edge cases. The utils.py
module was tested for image validation with valid images, corrupted files, and non-image files.
9.1.2 Integration Testing
Modules were tested together to verify seamless interaction. The encryption to email sending pipeline
was tested. The email receiving to decryption pipeline was tested. The GUI to encryption and
decryption to status updates pipeline was tested. Key generation to key hashing to encryption to
decryption consistency was verified.
9.1.3 Functional Testing
Each feature was tested for correct behavior. Image selection and validation was verified. Encryption
was confirmed to produce valid .enc files. Decryption with the correct key was verified to restore the
original image. Decryption with the wrong key was confirmed to produce an appropriate error message.
Email sending was verified to successfully transmit the encrypted file. Email receiving was confirmed
to correctly download .enc attachments.
9.1.4 System Testing
The complete application was tested end-to-end covering the full workflow from selecting an image
through generating a key, encrypting, configuring email, sending, receiving, and finally decrypting.
Testing was done with various image sizes from small thumbnails to large high-resolution images,
different image formats, and under different network conditions.
9.1.5 White Box Testing
Internal code paths were thoroughly examined. All branches in encryption and decryption logic were
tested including valid data, corrupted data, and empty data scenarios. PKCS7 padding edge cases were
tested for data exactly divisible by the block size. MIME filename decoding with various encodings was
verified. The .env file parsing with malformed inputs was tested. Error handling paths in all modules
were exercised.
9.1.6 Black Box Testing
The system was tested from the user's perspective without knowledge of internal implementation.
Users attempted to encrypt and decrypt images using only the GUI. Various invalid inputs were
provided including non-image files, empty keys, and invalid email addresses. The system's response to
each input was verified against expected behavior.
9.2 Test Strategy and Approach

The test objectives were to verify that AES-256 encryption correctly transforms image data into
unreadable ciphertext, that decryption with the correct key produces a bit-identical copy of the original
image, that decryption with an incorrect key fails gracefully with an informative error message, that
email sending and receiving work correctly, that the GUI remains responsive during heavy operations,
and that all error conditions are handled gracefully with user-friendly messages.
9.2.2 Features to be Tested
Features tested include image file selection and validation, AES-256-CBC encryption and decryption,
PKCS7 padding and unpadding, SHA-256 key derivation, cryptographically random key generation,
email configuration save and load, SMTP email sending with attachment, IMAP email receiving and
attachment download, GUI responsiveness and thread safety, progress bar and status updates, clipboard
operations, password visibility toggle, and error handling for all failure scenarios.

10. Conclusion
The project Secure Image Transfer System Using AES-256 Encryption successfully demonstrates how
AES-256 encryption, combined with secure email transmission protocols, can be effectively

Python-based approach provides excellent flexibility, cross-platform compatibility, and ease of use.
